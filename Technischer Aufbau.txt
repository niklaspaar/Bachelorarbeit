1. Motivation und Zielbild
Ziel ist es, den bestehenden Tserv-basierten Optimierungsablauf so zu kapseln und zu erweitern, dass:

mehrere Optimierer parallel laufen können
Ressourcen (CPU, RAM) kontrolliert werden
Anfragen asynchron über REST gestellt werden können
Warteschlangen & Fehlerszenarien sauber abgebildet sind
der Optimierer unabhängig vom Tserv-Prozess betrieben werden kann

➡️ Daraus ergibt sich ein klassisches Job-Scheduling- und Orchestrierungsproblem.

2. Framework des OptimSchedulers
2.1 Rolle des OptimSchedulers
Der OptimScheduler ist das zentrale Steuermodul zwischen:

Eingehenden Requests (REST)
Warteschlange
laufenden Optimierer-Instanzen
Systemressourcen

Er ist zustandsbehaftet, im Gegensatz zu den Optimierern selbst.
Kernaufgaben
Basierend auf deinen funktionalen Anforderungen:

3. Unabhängiger Betrieb vom Tserv
Das ist ein entscheidender Bachelorarbeits-Punkt.
3.1 Aktueller Zustand (vereinfacht)

Tserv:

baut Optimierer (make)
ruft Shell-Skripte auf
blockiert indirekt Ressourcen


Optimierer:

sind eigentlich standalone lauffähig
werden aber fest über Tserv gestartet




3.2 Zielzustand: Entkopplung
Prinzip

Tserv → Client
OptimScheduler → Service
Optimierer → Worker

➡️ Kommunikation nur über definierte Schnittstellen.

3.3 Architekturvariante A: Prozessbasiert (ohne Container)
Für die Bachelorarbeit gut geeignet, wenn du „nah am System“ bleiben willst.

OptimScheduler ist ein eigenständiger Dienst
Optimierer werden per:

fork/exec oder
std::system()
gestartet


Überwachung via:

Exit Codes
Laufzeit
Speicherverbrauch (/proc unter Linux)



✅ Vorteile

Einfach
Kein Docker-Zwang
Passt gut zu bestehendem Tserv-Code

❌ Nachteile

Schlechtere Isolation
Skaliert schlecht auf mehrere Maschinen


4. Docker-Container: Kapselung der Optimierer
Jetzt kommen Docker & Kubernetes sinnvoll ins Spiel.
4.1 Idee von Docker in deinem Szenario
Jeder Optimierer läuft in einem eigenen Container

Container enthält:

ftlopt binary
benötigte Libs
Startskript (doFtlopt.sh)


Input per Volume-Mount
Output per Volume oder JSON-Response

docker run \
  -v /data/job123:/work \
  ftlopt-image \
  ftlopt -i input.json -s output.json


4.2 Vorteile für deine Arbeit
✅ Saubere Ressourcengrenzen

RAM-Limit
CPU-Limit

✅ Reproduzierbarkeit

Gleiche Umgebung
Gleiche Bibliotheken

✅ Sicherheit

Kein direkter Zugriff auf Host

✅ Idealer Übergang zu Kubernetes

4.3 Rolle des OptimSchedulers mit Docker
Der Scheduler:

erstellt Container
überwacht Status
beendet Container bei Timeout
sammelt Ergebnisse

Du kannst das konzeptionell beschreiben, ohne Docker selbst implementieren zu müssen.
